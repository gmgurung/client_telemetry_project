# Telemetry Pipeline Presentation Guide

This document outlines key points for presenting the Telemetry Frustration Scoring Pipeline, based on the `Untitled Diagram.drawio.svg` architectural map. It covers the three main phases of the pipeline and the iterations made to each component.

---

## 🏗️ Phase 1: Data Ingestion & Preprocessing

**Key Presentation Points:**
*   **Source Agnostic:** The pipeline handles raw telemetry data in `.ndjson` format from both AWS S3 and local file systems.
*   **Schema-Driven:** Ingestion is controlled by `config/schema_default.json`, allowing the system to adapt to different telemetry formats (Web, Mobile, etc.) without code changes.
*   **Session-Level Aggregation:** The core of `preprocess.py` is grouping individual events (clicks, page views) into cohesive user sessions.
*   **Feature Engineering:** Raw events are transformed into 13 specific numerical features (e.g., rage click count, dwell time, error frequency) that quantify user frustration.

**Version Iterations:**
*   **V1:** Local-only processing of flat JSON files; fixed feature mapping.
*   **V2:** Added S3 integration and the abstraction of the `DataLoader` class for extensibility.
*   **V3 (Current):** Introduced the JSON-based schema mapping and engineered more complex behavioral features like `avg_inter_event_gap_ms`.

---

## 🧠 Phase 2: Model Training & Artifacts

**Key Presentation Points:**
*   **Ensemble Strategy:** Uses a two-model approach combining **Autoencoder** (deep learning) and **Isolation Forest** (tree-based) to identify anomalies from different perspectives.
*   **Automated Scaling:** Implements `StandardScaler` to ensure all 13 features are normalized before training, preventing high-magnitude features from dominating the model.
*   **Rigorous Validation:** The `Validation` block ensures that only models passing strict quality checks (e.g., discrimination std > 0.1, finite scoring) are promoted to production.
*   **Versioned Artifacts:** Successfully trained models are packaged into versioned tarballs (`model_vX.tar.gz`) and stored in a dedicated S3 bucket.

**Version Iterations:**
*   **V1:** Single Autoencoder model; manual validation.
*   **V2:** Introduced Isolation Forest to create an ensemble; added basic validation scripts.
*   **V3 (Current):** Added "Dynamic Threshold Tuning" using a semi-supervised feedback loop (tuning against labeled data) and fully automated S3 versioning.

---

## 🚀 Phase 3: Deployment & Inference

**Key Presentation Points:**
*   **Dual Inference Paths:**
    1.  **Real-Time:** SageMaker Endpoint for high-performance, scalable API requests.
    2.  **Edge/Local:** `client.py` can sync model artifacts locally for scoring without network overhead.
*   **Automated Triggers:** `polling_runner.py` monitors S3 raw data continuously, while a Lambda function provides event-driven triggers.
*   **"Brain Swap" Capability:** The `client.py` can automatically detect and download new model versions from S3 without downtime.
*   **Severity Classification:** The final 0-10 score is translated into human-readable levels (Normal, Medium, High) in the Dashboard.

**Version Iterations:**
*   **V1:** Script-based local inference only.
*   **V2:** First SageMaker endpoint deployment; manual model synchronization.
*   **V3 (Current):** Full automation with `polling_runner.py`, Lambda triggers, and the "Brain Swap" logic for seamless model updates.

---

## 📊 Summary of Pipeline Flow
1.  **Ingest:** `preprocess.py` transforms raw logs into `.csv` features.
2.  **Train:** `train.py` builds the ensemble and validates the "Brain".
3.  **Package:** `package.py` converts models to TensorFlow `SavedModel` format.
4.  **Deploy:** `deploy_endpoint.py` provisions the AWS SageMaker infrastructure.
5.  **Serve:** `serve.py` handles real-time scoring and classification for the end-user Dashboard.
